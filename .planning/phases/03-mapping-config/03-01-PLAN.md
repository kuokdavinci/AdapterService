<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MongoDB → ConfigCache | Cached configs come from DB; cache trusts DB data is valid |
| ConfigCache → ConfigLoader | Loader trusts cache entries are unexpired and uncorrupted |
| ConfigValidator → ConfigLoader | Validator output determines if config is safe to use |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | ConfigCache | mitigate | threading.Lock prevents concurrent write corruption; cache key is deterministic from config fields (not user-supplied) |
| T-03-02 | Spoofing | ConfigCache key | mitigate | Cache key derived from validated config fields (partner, workflow_type, file_type, version) — not from user input directly |
| T-03-03 | Repudiation | ConfigValidator | accept | Validation errors are returned as list, not logged; caller (ConfigLoader) is responsible for logging/auditing |
| T-03-04 | Information Disclosure | ConfigCache | accept | Cache stores MappingConfig which contains parsing rules, not credentials or PII; low-value target |
| T-03-05 | Denial of Service | ConfigCache | mitigate | Lazy cleanup on get prevents unbounded memory growth; no background threads; max entries bounded by unique config combinations |
</threat_model>
