**Logic:**
1. If value is None or empty string → return ValidationError("cannot map empty/null value for path '{fm.path}'")
2. Convert value to string: str_value = str(value)
3. Look up str_value in fm.mapping dict:

Add build_canonical method to TransactionNormalizer in src/normalizer/normalizer.py.

**Method signature:**
```python
@staticmethod
def build_canonical(
    data: dict[str, Any],
    errors: list[ValidationError],
    row_number: Optional[int] = None,
) -> tuple[CanonicalTransaction | None, list[ValidationError]]:
```

**Required fields for CanonicalTransaction:**
- `id` (str): must exist in data, must be non-empty string
- `amount` (Decimal): must exist in data, must be Decimal
- `currency` (str): must exist in data, must be non-empty string
- `status` (str): must exist in data, must be convertible to TransactionStatus enum

**Optional fields:**
- `trace` (str): include if present in data
- `transDate` (datetime): include if present in data
- Any other keys in data → placed in `extra` dict

**Logic:**
1. Check required fields exist in data:
   - Missing "id" → ValidationError(field="id", reason="required field 'id' not found in normalized data")
   - Missing "amount" → ValidationError(field="amount", reason="required field 'amount' not found in normalized data")
   - Missing "currency" → ValidationError(field="currency", reason="required field 'currency' not found in normalized data")
   - Missing "status" → ValidationError(field="status", reason="required field 'status' not found in normalized data")
2. If any required field missing → return (None, errors + new_errors)
3. Convert status string to TransactionStatus enum:
   - Try TransactionStatus(data["status"])
   - If ValueError (invalid enum value) → ValidationError(field="status", reason="invalid status value '{data['status']}' — must be one of SUCCESS, FAILED, PENDING, REVERSED")
   - If status error → return (None, errors + new_errors)
4. Build extra dict: {k: v for k, v in data.items() if k not in ("id", "trace", "amount", "currency", "status", "transDate")}
5. Construct CanonicalTransaction:
   ```python
   CanonicalTransaction(
       id=str(data["id"]),
       trace=str(data["trace"]) if "trace" in data else None,
       amount=data["amount"],
       currency=str(data["currency"]),
       status=TransactionStatus(data["status"]),
       transDate=data.get("transDate"),
       extra=extra,
   )
   ```
6. Return (transaction, errors) on success

**Do NOT:**
- Do NOT modify data dict (read-only)
- Do NOT silently skip missing required fields — always produce ValidationError
- Do NOT use pydantic validation errors — use ValidationError from src.core.types

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mapping dict → _convert_mapping | Partner-configured mapping values could contain unexpected keys or missing "others" |
| Normalized data → build_canonical | Normalized dict may have missing/invalid values for required CanonicalTransaction fields |
| Status string → TransactionStatus | Invalid status strings could cause enum construction failure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Tampering | _convert_mapping | mitigate | "others" fallback is opt-in (configured in MappingConfig); missing "others" produces explicit ValidationError — no silent defaulting to SUCCESS or FAILED |
| T-04-08 | Tampering | build_canonical status conversion | mitigate | TransactionStatus(data["status"]) raises ValueError for invalid values → caught and converted to ValidationError; no raw exception escape |
| T-04-09 | Tampering | build_canonical required fields | mitigate | Explicit check for id/amount/currency/status before construction; missing fields produce ValidationError with specific field name |
| T-04-10 | Repudiation | build_canonical extra dict | mitigate | Unknown fields placed in extra dict (not silently dropped); preserves all normalized data for audit trail |
| T-04-11 | Information Disclosure | ValidationError in build_canonical | mitigate | Error reasons reference field names and expected values, not raw data content; no PII leakage in error messages |
</threat_model>
