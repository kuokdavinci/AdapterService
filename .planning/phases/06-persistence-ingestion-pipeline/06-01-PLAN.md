  <behavior>
    - IngestionPipeline constructor accepts: db (AsyncIOMotorDatabase), config_loader (ConfigLoader), batch_size (int, default 100)
    - process_file(file_path, partner, workflow_type, file_type, reconciliation_date, config_version=None) → IngestionResult
    - IngestionResult dataclass: file_record (ReconciliationFile), stats (ProcessingStats), errors (list[dict])
    - Pipeline flow:
      1. Compute SHA256 hash of file_path
      2. Check file duplicate via reconciliation_file_repo.find_by_file_hash — if found, return early with error
      3. Create ReconciliationFile record with PROCESSING status
      4. Load MappingConfig via config_loader (by partner_type or by version)
      5. Create ExcelStreamReader via from_mapping_config
      6. Create TransactionNormalizer with config.field_mappings
      7. Create Validator with data_container_repo and reconciliation_file_repo
      8. For each row (enumerate from start_row):
         a. Convert tuple to dict keyed by column letter (A, B, C, ...)
         b. Normalize via normalizer.normalize(row_dict, row_number)
         c. If normalization errors exist → count as failed, collect error, continue
         d. Build CanonicalTransaction via TransactionNormalizer.build_canonical()
         e. If build fails → count as failed, collect error, continue
         f. Validate via validator.validate_with_duplicates(txn, identify=partner, reconciliation_date, file_hash, row_number, txn.trace)
         g. If valid → add to batch buffer
         h. If invalid → count as failed, collect error, continue
         i. When batch buffer reaches batch_size → flush to data_container_repo (bulk insert)
      9. Flush remaining batch buffer
      10. Update ReconciliationFile stats (total_rows, success_rows, failed_rows)
      11. Update ReconciliationFile status to COMPLETED
      12. Return IngestionResult with file_record, stats, errors
    - On any exception during processing: update status to FAILED, return partial stats
    - Batch insertion: use collection.insert_many for bulk inserts (more efficient than individual inserts)
    - Per-row error handling: never raise exceptions for bad rows — collect and continue
    - Column letter mapping: tuple index 0 → "A", 1 → "B", etc.
  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Excel file → IngestionPipeline | Untrusted file content, potentially malformed rows |
| IngestionPipeline → MongoDB | Batch insertions, status updates |
| ConfigLoader → IngestionPipeline | Config may be missing or invalid |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | process_file() | mitigate | Compute SHA256 file hash before processing; reject if hash matches existing file |
| T-06-02 | Denial of Service | ExcelStreamReader | mitigate | Use streaming read-only mode (openpyxl); never load entire file into memory |
| T-06-03 | Information Disclosure | IngestionResult errors | mitigate | Errors contain row numbers and field names only — no raw partner data in error messages |
| T-06-04 | Repudiation | Processing stats | mitigate | ReconciliationFile record created before processing begins; stats updated atomically |
| T-06-05 | Elevation of Privilege | DataContainerRepository.insert_many | mitigate | Validate each DataContainer via pydantic before batch insert; reject malformed documents |
| T-06-06 | Spoofing | ConfigLoader integration | mitigate | ConfigLoader validates config before return; ConfigLoadError caught and pipeline status set to FAILED |
</threat_model>
